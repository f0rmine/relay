# Relay Messenger

Relay — це месенджер реального часу з бекендом на FastAPI та фронтендом на Ionic Vue + Capacitor, який насамперед орієнтований на Android. У ньому є постійне зберігання даних у базі, JWT-автентифікація, приватні чати один на один, історія повідомлень, вкладення, сповіщення про прочитання, статус присутності, індикатори набору тексту, аватари, локальні сповіщення й FCM, Docker-розгортання та тести бекенду. На перший погляд — звичайний месенджер. Але всередині тут зібрано доволі багато прикладної інфраструктури.

## Технологічний стек

- Бекенд: Python, FastAPI, Pydantic v2, SQLAlchemy 2.0 async, Alembic
- База даних: PostgreSQL як постійне джерело достовірних даних
- Обмін даними в реальному часі: нативні WebSocket у FastAPI
- Допоміжний компонент для обміну даними в реальному часі: Redis для статусу присутності, стану набору тексту та тимчасових з'єднань/розсилання через pub/sub
- Мобільний застосунок: Ionic Vue, Vue 3 Composition API, TypeScript, Vite, Pinia, Vue Router
- Android: Capacitor
- Розгортання: Docker Compose із PostgreSQL, Redis, бекендом і постійним сховищем завантажених файлів

## Архітектура

FastAPI надає REST API та шлюз WebSocket. PostgreSQL зберігає користувачів, розмови, учасників, зашифровані дані повідомлень, сповіщення про прочитання, токени оновлення, токени скидання пароля та метадані вкладень. Текст нових повідомлень і байти вкладень чату шифруються за допомогою AES-256-GCM перед записом у сховище, а розшифровуються тільки після авторизації учасника. Просто? Так, але саме ця межа між зберіганням і доступом тут важлива.

Файли лежать у постійному локальному томі завантажень: аватари віддаються публічно, щоб профілі нормально відображалися, а зашифровані вкладення чату проходять через автентифіковані маршрути завантаження. Redis, своєю чергою, тримає лише тимчасовий стан реального часу: онлайн-присутність, індикатори набору тексту та розсилання через pub/sub для майбутнього запуску кількох екземплярів бекенду. Постійні дані чату в Redis не зберігаються.

## Налаштування бекенду

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
cp .env.example .env
```

Згенеруйте 32-байтовий ключ шифрування та додайте його значення у форматі Base64 до `ENCRYPTION_KEYS` перед запуском бекенду:

```bash
python3 -c "import base64,secrets; print(base64.b64encode(secrets.token_bytes(32)).decode())"
```

Приклад структури конфігурації. Значення нижче навмисно залишено як заповнювач:

```env
ENCRYPTION_ACTIVE_KEY_ID=v1
ENCRYPTION_KEYS={"v1":"REPLACE_WITH_BASE64_ENCODED_32_BYTE_KEY"}
```

Запустіть міграції:

```bash
alembic upgrade head
```

Запустіть бекенд:

```bash
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000 --no-access-log
```

Необов'язкові тестові користувачі:

```bash
python -m app.scripts_seed
```

## Налаштування Docker

З кореневого каталогу репозиторію:

```bash
cp .env.example .env
docker compose up -d postgres redis
docker compose build backend
docker compose run --rm backend alembic upgrade head
docker compose up backend
```

API бекенду: `http://localhost:8000`

Порти PostgreSQL і Redis на хості прив'язані до `127.0.0.1` лише для локальної діагностики. Контейнери взаємодіють через приватну мережу Compose; порти `5432` або `6379` не варто відкривати для LAN чи інтернету.

Під час запуску бекенд перевіряє активний ключ шифрування. Відсутні ключі, невідомі ідентифікатори активних ключів, некоректний Base64 і ключі, довжина яких не дорівнює точно 32 байтам, зупиняють старт застосунку замість тихого зберігання відкритого тексту. Дрібниця на вигляд, але корисна.

### Ротація ключів шифрування

Згенеруйте новий 32-байтовий ключ, залиште попередній ключ у JSON-мапі та змініть лише активний ідентифікатор:

```env
ENCRYPTION_ACTIVE_KEY_ID=v2
ENCRYPTION_KEYS={"v1":"OLD_BASE64_KEY","v2":"NEW_BASE64_KEY"}
```

Після перезапуску нові повідомлення та вкладення використовуватимуть `v2`; наявні записи `v1` залишаться доступними для розшифрування. Не видаляйте `v1`, доки кожен запис, що посилається на нього, не буде повторно зашифрований або видалений. Варто зазначити й інше: мапу ключів потрібно резервувати окремо від PostgreSQL і тому завантажень, бо втрата потрібного ключа назавжди закриває доступ до відповідного зашифрованого вмісту.

Постійні томи:

- `postgres_data`: дані PostgreSQL
- `redis_data`: дані Redis у режимі append-only
- `uploads`: завантажені аватари та вкладення

## Налаштування мобільного застосунку

Потрібні Node.js 20.19+ або 22.12+ та npm.

```bash
cd mobile
npm install
cp .env.example .env
npm run dev
```

Для розробки у браузері використовуються:

```env
VITE_API_BASE_URL=http://localhost:8000
VITE_WS_BASE_URL=ws://localhost:8000
```

У робочому середовищі HTTPS/WSS використовує ті самі змінні, без змін у коді:

```env
VITE_API_BASE_URL=https://relay.example.com
VITE_WS_BASE_URL=wss://relay.example.com
```

Завершуйте TLS на Caddy, Nginx, Tailscale Serve або іншому довіреному зворотному проксі, а потім переспрямовуйте трафік REST і `/ws` до FastAPI на порті `8000`. `http/ws` краще лишати тільки для локальної розробки або довіреного зашифрованого тунелю.

Додайте публічне джерело фронтенду, наприклад `https://relay.example.com`, до `CORS_ORIGINS` бекенду. Під час запуску фронтенду URL-адреси API та WebSocket відхиляють непідтримувані схеми, рядки запиту та фрагменти.

Для емулятора Android зазвичай потрібні:

```env
VITE_API_BASE_URL=http://10.0.2.2:8000
VITE_WS_BASE_URL=ws://10.0.2.2:8000
```

Для справжнього телефона Android у тій самій мережі Wi-Fi/LAN потрібна LAN IP-адреса хоста, наприклад:

```env
VITE_API_BASE_URL=http://192.168.1.50:8000
VITE_WS_BASE_URL=ws://192.168.1.50:8000
```

### Зберігання токенів в Android

У нативному Android токени доступу й оновлення, а також кешований профіль користувача зберігаються плагіном безпечного сховища Capacitor із використанням ключа AES-GCM, захищеного Android Keystore. Під час розробки у браузері використовується резервний варіант із `localStorage`, бо браузер не має доступу до Android Keystore. Під час першого нативного запуску після оновлення застарілі значення автентифікації переносяться з `localStorage` до безпечного сховища.

Перевірка вручну:

1. Зберіть і встановіть застосунок Android, увійдіть, закрийте та знову відкрийте його; сеанс має відновитися.
2. Перевірте сховище WebView через Chrome DevTools; `accessToken`, `refreshToken` та `authUser` не мають бути в `localStorage` на Android.
3. Вийдіть і перевірте, що після повторного відкриття застосунок вимагає входу.
4. Запустіть браузерну збірку та перевірте, що вхід/оновлення й далі працюють через задокументований резервний варіант для вебу.

## Сповіщення

Застосунок використовує два шляхи сповіщень:

- Локальні сповіщення: коли застосунок працює й отримує через WebSocket подію `message:new` поза відкритим у цей момент чатом.
- Push-сповіщення FCM: коли застосунок одержувача закритий/від'єднаний, а бекенд зберігає нове повідомлення.

Для сповіщень Android, коли застосунок закрито:

1. Створіть проєкт Firebase і застосунок Android з іменем пакета `com.relay.messenger`.
2. Завантажте `google-services.json` із Firebase і розмістіть його в `mobile/android/app/google-services.json` після виконання `npx cap add android`.
3. Створіть JSON облікового запису служби Firebase у Firebase Console і розмістіть його поза git, наприклад `firebase-service-account.json`.
4. У файлі `.env` бекенду задайте:

```env
PUSH_NOTIFICATIONS_ENABLED=true
FIREBASE_PROJECT_ID=your-firebase-project-id
FIREBASE_SERVICE_ACCOUNT_FILE=/absolute/path/to/firebase-service-account.json
```

Для Docker розмістіть ігнорований файл облікового запису служби в `backend/firebase-service-account.json`, задайте `FIREBASE_SERVICE_ACCOUNT_FILE=/app/firebase-service-account.json` і додайте перевизначення Firebase:

```bash
docker compose -f docker-compose.yml -f docker-compose.firebase.yml up -d backend
```

`./scripts/backend-start.sh` автоматично додає це перевизначення, якщо існує `backend/firebase-service-account.json`. Справжній JSON-файл не додавайте до git.

Якщо в журналах бекенду з'являється відмова в дозволі `cloudmessaging.messages.create`, надайте обліковому запису служби роль Firebase Cloud Messaging API Admin (`roles/firebasecloudmessaging.admin`) у Google Cloud IAM.

Щоб перевірити, який обліковий запис служби та проєкт Firebase використовує бекенд:

```bash
./scripts/firebase-check.sh
```

Скрипт виводить лише безпечні метадані, як-от `project_id` і `client_email`, а також точну команду IAM для `gcloud`. Якщо обліковий запис служби ще не відображається в Google Cloud IAM, скористайтеся **Grant access**, вставте виведений `client_email` як нового суб'єкта та призначте **Firebase Cloud Messaging API Admin**.

## Події WebSocket

Під'єднайтеся через `ws://HOST:8000/ws` у локальному середовищі розробки або через `wss://relay.example.com/ws` у робочому середовищі, а потім надішліть перший фрейм у такому вигляді:

```json
{"type":"auth","payload":{"token":"ACCESS_TOKEN"}}
```

Від клієнта до сервера:

- `auth`
- `conversation:join`
- `conversation:leave`
- `message:send`
- `message:delete`
- `message:read`
- `typing:start`
- `typing:stop`

Від сервера до клієнта:

- `message:new`
- `message:deleted`
- `message:read`
- `conversation:updated`
- `user:online`
- `user:offline`
- `typing:update`
- `error`

Вкладення спочатку завантажуються, а вже потім доставляються учасникам у `message:new`, коли відправник надсилає повідомлення. Завантаження, не пов'язані з повідомленнями, не транслюються.
Для попереднього перегляду та завантаження вкладень використовуються автентифіковані запити `/attachments/{attachment_id}/download`; необроблені файли вкладень не доступні через публічний статичний маршрут завантажень.

## Схема бази даних

Основні таблиці:

- `users`: ідентифікаційні дані, електронна адреса, хеш пароля, аватар, час останньої активності
- `refresh_tokens`: хешовані постійно збережені токени оновлення
- `password_reset_tokens`: хешовані токени скидання
- `conversations`: записи приватних чатів один на один
- `conversation_participants`: записи учасників і час прочитання на рівні розмови
- `messages`: сумісність із застарілим відкритим текстом, а також шифротекст AES-GCM, одноразове значення, ідентифікатор ключа, версія шифрування, часові мітки та метадані м'якого видалення
- `message_reads`: сповіщення про прочитання окремих повідомлень
- `attachments`: метадані завантаженого файла, шлях до зашифрованого сховища, одноразове значення AES-GCM, ідентифікатор ключа та версія шифрування
- `device_tokens`: токени пристроїв Android/FCM та вибрані для них мови сповіщень, коли застосунок закрито

Застосунок використовує рядки UUID як ідентифікатори. PostgreSQL є джерелом достовірних даних для всіх постійних даних.

## Ключі Redis

- `presence:user:{user_id}`: короткочасна позначка присутності онлайн
- `typing:conversation:{conversation_id}:user:{user_id}`: короткочасна позначка набору тексту
- `pubsub:broadcast`: канал розсилання подій WebSocket

Дані Redis тимчасові, і їх можна безпечно втратити.

## Примітки щодо безпеки

- Паролі хешуються за допомогою Argon2.
- Токени доступу й оновлення використовують різні секрети.
- Токени оновлення зберігаються в PostgreSQL у хешованому вигляді.
- Нативний Android зберігає токени доступу й оновлення в безпечному сховищі, захищеному Android Keystore. Під час розробки у браузері використовується резервний варіант із `localStorage`; у публічному браузерному розгортанні токени оновлення варто перенести до безпечних cookie HttpOnly для кращої ізоляції від XSS.
- Текст нових повідомлень і байти вкладень шифруються під час зберігання за допомогою AES-256-GCM з унікальним випадковим одноразовим значенням для кожного запису.
- Ключі шифрування версіонуються. Нові записи використовують `ENCRYPTION_ACTIVE_KEY_ID`; старі ідентифікатори ключів мають залишатися в `ENCRYPTION_KEYS`, доки відповідні записи не буде повторно зашифровано.
- Чутливі до автентифікації кінцеві точки мають простий внутрішньопроцесний обмежувач частоти запитів для кожної IP-адреси.
- Маршрути та дії WebSocket перевіряють поточного користувача.
- Лише учасники розмови можуть читати, надсилати, видаляти або позначати повідомлення як прочитані.
- Для завантажень використовуються згенеровані імена файлів; дозволено лише зображення та поширені MIME-типи/розширення документів.
- Файли аватарів публічні, щоб їх можна було відображати в профілях; для доступу до файлів вкладень чату потрібен автентифікований маршрут.
- Текст повідомлень відображається у Vue як звичайний текст, а не як небезпечний HTML.
- Docker запускає Uvicorn із вимкненим журналюванням доступу, тому JWT-токени в рядках запиту WebSocket не потрапляють до звичайних журналів контейнера.
- Для публічних розгортань запускайте бекенд за HTTPS/WSS і ротуйте ключі JWT та шифрування даних.

Це серверне шифрування під час зберігання, а не наскрізне шифрування: запущений бекенд може розшифровувати авторизовані відповіді та попередній вміст push-сповіщень. Дампи бази даних і скопійовані томи завантажень не містять відкритого тексту нових збережених повідомлень/файлів, але компрометація і бекенду, і його ключів шифрування може розкрити вміст.

## Самостійне розгортання

ПК з Windows і Docker Desktop:

1. Встановіть Docker Desktop.
2. Скопіюйте `.env.example` до `.env`.
3. Виконайте наведені вище команди Docker.
4. Використовуйте `localhost` для тестування у браузері, `10.0.2.2` для емулятора Android або LAN IP-адресу ПК для справжнього телефона.

Сервер Linux із Docker Engine:

1. Встановіть Docker Engine і плагін Compose.
2. Скопіюйте `.env.example` до `.env` і задайте надійні секрети.
3. Виконайте міграції та запустіть стек.
4. Використовуйте домен із Caddy/Nginx та HTTPS або VPN, наприклад Tailscale, для приватного доступу.

Регулярно створюйте резервні копії PostgreSQL і зберігайте завантажені файли в постійному сховищі.

## Відомі обмеження

- Скидання пароля повертає токен через API замість надсилання електронного листа через SMTP.
- Для push-сповіщень FCM потрібно налаштувати проєкт Firebase; за замовчуванням вони вимкнені у `.env`.
- Для сповіщень, коли застосунок закрито, потрібні облікові дані Android/Firebase; локальні сповіщення WebSocket працюють без Firebase.
- Групові чати, голосові повідомлення, редагування повідомлень, реакції, блокування/скарги та адміністраторська модерація не реалізовані.
- Локальні завантаження придатні для локальних або невеликих самостійних розгортань; S3-сумісне об'єктне сховище буде кращим удосконаленням для більших розгортань.
- Повне наскрізне шифрування навмисно залишено як майбутнє вдосконалення.
- Міграція `0003_content_encryption` додає необов'язкові метадані шифрування, тому застарілі записи/файли з відкритим текстом залишаються доступними для читання без порушення історії. Вона не перезаписує наявний вміст; виконайте контрольоване заповнення даних, перш ніж заявляти, що кожен історичний рядок/файл зашифровано.
